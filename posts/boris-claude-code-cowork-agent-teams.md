Claude Code 之父、Anthropic 工程负责人 **Boris**，在最近一期播客里把很多关键信息讲得非常透。

从 Claude Code 最初只是一个终端聊天工具，到后面为什么会接上 bash、文件编辑，再一路演进成今天的 agent 形态；从 Co-work 为什么能在 **10 天** 里做出来，到它真正复杂的地方为什么是 **安全性**；从 Agent Teams 的底层魔法，到 Anthropic 内部为什么几乎不写 PRD、而是疯狂做原型——这场访谈的信息密度高得离谱。

如果你想知道顶尖 AI 产品团队到底怎么想产品、怎么做工程、怎么理解多智能体，这篇很值得认真看完。

---

## 核心看点速览

- Claude Code 的起点其实不是「编程 agent」，而是一个 **命令行聊天机器人**
- 真正的转折点，是 Boris 给模型接上了第一个工具：**bash**
- Claude Code 的核心架构并不玄乎，本质上就是：**agent 循环 + 执行系统 + 一整圈安全机制**
- Anthropic 一度认真讨论过：这么强的工具要不要只留在内部用
- Co-work 虽然看起来像个「套壳产品」，但真正难的是 **安全护栏、权限设计和虚拟机环境**
- Agent Teams 最关键的概念，不是「多 agent」，而是 **非相关上下文窗口**
- Anthropic 内部几乎不写 PRD，很多功能是通过 **几十上百个原型** 试出来的
- Boris 认为 2026 会是 **通才之年**，甚至半开玩笑地说会是 **“ADHD 之年”**

---

## 一、加入 Anthropic 后，他的第一个 PR 直接被拒了

Boris 说，他刚加入 Anthropic 时，写的第一个 pull request 是 **完全手写代码** 完成的。

在他当时的理解里，「写代码」本来就该这么做。但他的入职导师 Adam Wolf 看完之后，直接把 PR 打回来了，理由非常简单：

> 你应该用 Clyde 来做这个。

这里的 **Clyde**，其实就是 Claude Code 的前身。那个工具很原始，用 Python 写的，启动甚至要 40 秒左右，本质上更像一个研究工具，也还没有今天这种成熟的 agent 能力。

但 Boris 折腾了半天学会之后，模型真的替他生成了一个 **能工作的完整 PR**。这成了他在 Anthropic 的第一个「AI 震撼时刻」。

因为在那之前，大多数工程师对 AI 编码的理解仍然停留在 **IDE 补全**、**tab completion** 这个层级。Boris 也是第一次真正意识到：模型不是只会补几行代码，它已经可以直接交出一个能跑通的完整改动。

---

## 二、Claude Code 最开始，其实就是个 ChatBot

很多人今天回头看 Claude Code，会默认它一开始就是为写代码而设计的 agent。

但 Boris 说，完全不是。

最早的时候，他只是想快速试试 Anthropic 的 API，又不想碰 UI，于是就写了一个非常简单的小工具：

- 在终端里输入问题
- 调 Anthropic API
- 模型回一段文字

说白了，就是一个 **命令行聊天机器人**。

这也很符合当时大家对 AI 的认知：AI = ChatBot。

Boris 说，工程师通常是最早一批感知到形态变化的人。当 AI 从「对话式 AI」逐渐转向「Agent AI」时，工程师会比非工程师更早意识到变化已经发生了。但如果你今天去问很多普通用户「AI 是什么」，他们仍然会回答：**就是聊天机器人。**

这也是为什么 Boris 会对后来的 **Co-work** 特别兴奋：因为它本质上是在把工程师先看到的 agent 能力，包装成一个更容易被非工程师理解和使用的产品。

---

## 三、从 ChatBot 到 Agent：真正的转折点是 bash

Claude Code 的真正拐点，不是模型突然会写代码了，而是 Boris 做了一个很简单但意义巨大的实验：

> 给模型接上工具。

第一个工具就是 **bash**。

Boris 当时自己也不确定 bash 到底能干嘛，于是就随手问了模型一个问题：

> 我现在在听什么音乐？

结果模型自己写了一个小的 AppleScript，再配合命令行工具去打开播放器、读取当前播放信息，最后把答案给出来了。

这个任务是用 **Sonnet 3.5** 一次做完的。

Boris 说，这给了他第二次「AI 震撼时刻」。因为他突然意识到：

> **模型天然就想使用工具。**

只要你给它工具，它会主动去理解工具、调用工具，然后把任务做完。

而这也让他彻底放弃了另一种旧思路：

- 不是把模型当成传统程序里的一个「智能模块」
- 而是把模型本身当成一个独立系统
- 你要做的是给它环境、给它工具、给它边界
- 然后让它自己去工作

这和 AI 研究里那条经典的 **“Bitter Lesson（苦涩的教训）”** 很像：很多时候，真正有效的方法不是人为写死规则，而是让系统自己发挥能力。

---

## 四、Claude Code 发布前，Anthropic 甚至想过只给内部用

Claude Code 在内部变强以后，Anthropic 认真讨论过一个问题：

> 这么强的工具，要不要干脆只在内部使用？

因为它确实给工程团队带来了非常明显的效率提升。

但最后他们还是决定发布。Boris 给出的原因很 Anthropic：

> **因为这样可以在真实世界里研究安全问题。**

在 Anthropic 看来，安全从来不是一个抽象口号，而是一个分层展开的研究问题：

- 底层是 **alignment（对齐）** 和 **mechanistic interpretability（机制可解释性）**
- 中层是 **evals（评测）**
- 再往上，是把模型真正放进现实环境，看用户到底怎么用、会遇到什么风险、会暴露什么问题

也正是在真实环境里，Claude Code 的增长曲线开始夸张到几乎垂直。

Boris 回忆说，当时内部评审会上，Dario 看到使用率曲线之后，甚至直接问他：

> 你是不是强制大家用了？

而 Boris 的回答是：没有，我们只是做了个工具，大家在用脚投票。

今天在 Anthropic，几乎 **100% 的技术员工** 每天都在用 Claude Code，非技术员工的使用率也在迅速上升。

---

## 五、真正的转折点：Opus 4.5 让他几乎不再打开 IDE

Boris 提到，自己工作方式真正发生突变，是在 **Opus 4.5** 出现之后。

那时模型还没正式发布，但他们已经在内部 dogfooding。Boris 很快意识到一个事实：

> 自己几乎不需要再打开 IDE 了。

他甚至有段时间把 IDE 都卸掉了。不是仪式感，而是真的用不上。

后来他去欧洲旅行，一个月里提交了 **10 到 20 个 PR/天**，而且这些 PR：

- 几乎全是 Claude Code + Opus 4.5 写的
- 他自己一行都没改
- 一个月统计下来，模型只引入了 **两处 bug**

Boris 的原话很有意思：

> 如果这些代码是我自己写的，大概会有二十个 bug。

这也是很多工程师在 2025-2026 这一波里共同经历的拐点：

不是 AI 偶尔帮你写点代码，
而是你开始认真怀疑：

> **我还要不要亲手写这些？**

---

## 六、Boris 自己的工作流：五个 terminal 标签页 + plan mode

聊到工作流时，Boris 强调了一点：

> Claude Code 没有唯一正确的用法。

他们从一开始就故意把这个产品做得很 **hackable（可改造）**，因为工程师的工作流天生就高度个性化。

但 Boris 自己的默认工作流很典型：

- 同时开 **五个 terminal 标签页**
- 每个标签页都对应一个代码仓库 checkout
- 轮流启动 Claude Code
- 几乎每次都先进入 **plan mode**
- 如果本地终端不够用，就把任务「溢出」到桌面端或 Web 端继续跑

后面他们又逐步引入了 **Git worktree** 支持，让并行任务的隔离更自然。

Boris 特别喜欢这一点，因为它让多 agent 并行跑起来时，彼此互不污染。

另一个他自己都没预料到的变化，是 **iOS App**。

他说，自己现在经常早上醒来第一件事，就是在手机上先启动几个 agent。放在半年前，他自己都不会相信未来会有 **三分之一甚至一半的代码是在手机上写出来的**。

---

## 七、熟悉代码库后，他不再盯细节，而是盯 plan

Boris 把开发过程粗略分成两种模式：

### 1）刚进入新代码库
这个阶段最重要的是理解系统，所以他会更推荐：

- Learn Mode
- Explanatory 风格
- 让模型多解释、多展开、多讲清楚

### 2）已经熟悉代码库
这个阶段目标就变了，不再是学习，而是：

- 更高的效率
- 更多的交付
- 更快地把功能做出来

在这个阶段，他不会再盯住每个细节，而是会先让 Claude 进入 **plan mode**，把计划打磨好。

Boris 觉得，从 **Opus 4.5** 到 **4.6**，模型最大的变化之一就是：

> **只要计划足够对，它越来越接近 one-shot implementation。**

也就是说，真正的工作重点慢慢从「手写实现」转移到了「把 plan 调对」。

---

## 八、代码评审怎么做？Claude 先审，人最后拍板

当一个工程师一天能发 20 到 30 个 PR 时，一个不可回避的问题就是：

> **代码评审怎么办？**

Boris 的思路非常清晰：

### 在 AI 之前
他会把重复出现的问题记到表格里，累计到一定程度后，直接把它变成 lint 规则。

### 在 AI 之后
这套思路没有变，只是自动化程度更高了。

现在在 Anthropic，大致流程会是：

1. Claude Code 写代码时会主动跑测试
2. 有时还会自己补测试
3. 对 Claude Code 本体做改动时，它甚至会把自己拉起来做自检
4. 每个 PR 在 CI 里还会先经过 Claude 的一轮 code review
5. Claude 能自动修的就自动修
6. 修不了或者不确定的，留给人类 reviewer
7. 最后仍然必须由 **人类批准** 才能进系统

Boris 说得很坦白：

> LLM 天生是不确定的，所以一定要在流程里引入确定性机制。

这些确定性机制包括：

- 类型检查器
- lint
- 完整 build
- 自动测试
- 最终人工审查

也就是说，Anthropic 并不是「让 Claude 接管一切」，而是在尽可能把模型的能力和可验证的工程纪律叠在一起。

## 九、Claude Code 的架构，其实核心就是一个循环

Boris 对 Claude Code 架构的总结很简洁：

> 本质上就是一个 **query loop**，再加上一些工具。

如果再展开一点，可以理解成三部分：

- 一个核心的 **agent 循环**
- 一个端到端的 **执行系统**
- 一整圈围绕安全搭起来的 **防护层**

其中最复杂的其实不是 agent 本身，而是安全：

- 用户机器上的操作安全
- 模型潜在风险用法的监控
- prompt injection 防护
- 权限边界
- 人类在关键步骤中的介入

Boris 用的是典型的 **“瑞士奶酪模型”**：

没有哪一层防护是完美的，但只要层数够多，整体风险就能被显著压低。

## 十、Co-work 为什么能 10 天做出来？因为它本质上站在 Claude Code 之上

Co-work 在外界看来很像一个「把 Claude 套了个壳」的新产品。

但 Boris 的解释很有意思：

> 从产品逻辑上说，它确实没有外界想象得那么复杂。

原因是 Co-work 本质上就是 Claude 桌面应用的一部分。Claude Desktop 本来就有多个标签页，Co-work 只是其中一个新入口，而底层实际跑的仍然是 **Claude Code / Agent SDK** 那套东西。

所以它之所以能在 **10 天左右** 做出来，并不是因为团队会变魔术，而是因为：

- 基础能力已经存在
- 多终端能力已经存在
- agent 执行系统已经存在
- 剩下的问题，是把它重组为一个更适合非工程师的形态

但真正麻烦的地方，不在产品层，而在 **安全层**。

---

## 十一、Co-work 真正复杂的地方，不是功能，而是安全

Boris 反复强调，Co-work 的复杂性主要来自于：

> **它面对的是非技术用户。**

如果工程师在终端里删错了文件，那是工程师自己的风险；
但如果一个普通用户打开 Co-work，一不小心把全家照片删了，这就是产品设计灾难。

所以 Co-work 的复杂部分主要集中在这些地方：

- 大量 guardrails
- 运行在虚拟机中的隔离环境
- 更严格的权限模型
- Chrome 插件协作时的浏览器权限设计
- 隐私保护下的可观测性与监控

Boris 举的例子很典型：

他会让 Co-work 每周打开 Google 表格和 Slack，自动检查团队任务状态，再给对应工程师发提醒。这种体验对普通用户来说很强大，但如果没有一整套权限和安全设计托底，它也会变得非常危险。

也正因为此，Boris 说 Co-work 真正难的地方并不是「是不是套壳」，而是：

> **如何把 agent 能力变成非技术用户也敢放心使用的产品。**

---

## 十二、Agent Teams 的关键，不是“多个 agent”，而是“非相关上下文窗口”

这是整场访谈里最有价值的一个概念之一。

Boris 说，Anthropic 内部对 Agent Teams 的真正理解，不是简单地把一个 agent 变成多个 agent，而是：

> **非相关上下文窗口（non-correlated context windows）**

什么意思？

如果模型在同一个上下文里连续做两个任务，那么第二个任务天然会带着第一个任务的痕迹。

但如果主 agent 把一个任务丢给子 agent，而子 agent 在一个 **全新的、独立的上下文窗口** 里工作，那么它不会继承父上下文的包袱。

这会带来一个很有意思的效果：

- 有些任务需要共享上下文
- 但另一些任务，反而需要 **上下文彼此独立**

而当上下文彼此独立时，往问题里投入更多 token，结果反而可能更好。

Boris 觉得，这在某种意义上就是一种 **test-time compute**：

不是单纯堆更多模型参数，
而是在推理阶段用更多独立上下文去探索问题空间。

他们从去年 9、10 月左右就在实验这个方向，一直到 **Opus 4.6**，模型才真正开始学会利用这种结构。

---

## 十三、Agent Teams 为什么现在才发？因为它真的很烧 token

Boris 说，Teams 现在虽然已经能明显提升复杂任务效果，但它依然是一个相对谨慎发布的研究预览，原因很现实：

> **这东西本质上是在同时跑很多个 Claude。**

这意味着：

- 更复杂的结构
- 更高的 token 开销
- 更强的任务分发能力
- 但也不是每个任务都值得上 Teams

对于简单任务，单 agent 反而更合适。

但对于那些单个 Claude 已经明显吃力的问题，swarm / teams 结构就会非常有价值。

Boris 甚至提到，Anthropic 内部已经有不少功能，是直接通过 swarm 方式做出来的。

---

## 十四、Anthropic 内部为什么几乎不写 PRD？因为他们更信原型

这大概是整场访谈里最让产品人羡慕、也最容易让传统组织震惊的一段。

Boris 说，Anthropic 内部很多团队：

- 不太写 PRD
- 不依赖强制工单系统
- 更偏好直接做原型
- 通过原型去验证感觉和方向

为什么？

因为在今天这个阶段，问题往往不是「我们知不知道怎么实现」，而是：

> **我们甚至不知道正确答案长什么样。**

过去的软件开发更像是：

- 构建成本高
- 改方向成本更高
- 所以必须先写文档、对齐、再谨慎落地

但现在不同了。AI 让构建成本大幅下降，难点不再是能不能做出来，而是：

> **到底什么值得做、什么体验才对。**

因此 Anthropic 内部很多功能会先做：

- 十几个原型
- 几十个原型
- 有时甚至上百个原型

Boris 提到他们为一个待办事项系统做过十几二十个原型，为一个文件读取压缩视图做过三十个原型，而 Agent Teams 这种功能，更是花了几个月反复试了上百个版本，才找到合适的形态。

他说得非常直白：

> 如果从 Figma 静态图或者 PRD 开始，这些功能根本做不出来。

因为只有真正 build 出来、真正用过，你才知道体验到底对不对。

---

## 十五、Anthropic 内部几乎所有人都在写代码

Boris 还提到一个很有代表性的现象：

今天在 Anthropic，写代码已经不再是工程师的专属行为。

- 工程师写代码
- 工程经理写代码
- 设计师写代码
- 数据科学家写代码
- 财务同事也写代码

这背后不是因为所有人都突然转行做工程师了，而是因为 Claude Code 把写代码这件事的门槛打得足够低。

你不需要彻底理解一个庞大代码库，仍然可以借助 Claude Code 做出有价值的小改动，甚至顺便把工作流里的一些事自动化掉。

这也让 Boris 看到一个更大的趋势：

> AI 不只是提升工程师生产力，而是在 **模糊不同职能之间的边界**。

---

## 十六、我们正处在一个“AI 版的印刷机时代”

访谈后半段，Boris 对 AI 时代的一个比喻非常精彩：

> **Printing Press（印刷机）**

在印刷机出现之前，书写和复制知识是一种被少数人垄断的技能；
印刷机出现后，原来由少数人控制的能力，被迅速向更大范围开放。

Boris 觉得，今天的软件工程和智能能力，也正在经历类似的结构性变化：

- 原本只有少数专业工程师能做的事
- 现在越来越多普通人也能做
- 这会让旧的职业身份被重塑
- 也会催生全新的角色和分工

在这个逻辑下，他越来越看重的不是某一门具体技术，而是这些能力：

- 保持好奇心
- 愿意跨出自己的泳道
- 拥有假设驱动的工作方式
- 能把工程、产品、商业、设计等不同领域连接起来

所以他说，**2026 可能会是“通才之年”**。

---

## 十七、为什么他说 2026 可能是“ADHD 之年”

Boris 在访谈里还抛出了一个很抓人的说法：

> **2026 可能会是 “ADHD 之年”。**

当然这不是医学意义上的判断，而是一种工作方式观察。

他的意思是：

过去我们推崇的理想工作状态，是长时间深度专注、在单一问题上持续沉浸。

但今天很多高产 AI 工作者的日常，已经变成了：

- 同时管理多个并行任务
- 不断在不同上下文之间快速切换
- 在多个 agent、多个终端、多个产品形态中跳转
- 适应下一代模型一发布，工作方式就随之变化

这并不意味着深度思考不重要，而是说：

> **系统开始奖励“快速切换上下文”的能力。**

你要既能深度专注，
又能迅速切回另一个任务，
还得能根据新模型重新组织自己的工作方式。

这是一种全新的适应力要求。

---

## 十八、Boris 最后给出的判断：放下对语言、框架和代码风格的执念

如果把这场访谈压缩成一句对工程师最重要的话，可能就是这句：

> **放下对代码风格、语言、框架的执念。**

在 Boris 看来，这些传统争论正在快速失去重要性。因为模型可以使用任何语言和框架，如果你不喜欢，它甚至可以直接帮你重写。

真正仍然重要、甚至会变得更重要的，是：

- 方法论
- 结构化思考
- 调试能力
- 假设驱动的工作习惯
- 对复杂系统的好奇心
- 把不同领域联结起来的能力

也就是说，未来更稀缺的，不是「会写某种代码的人」，而是：

> **知道该做什么、为什么这么做、以及如何组织复杂问题的人。**

---

## 结语

这场播客最有意思的地方，不在于它又一次证明了 Claude Code 很强，也不在于 Boris 说了多少让人兴奋的产品细节。

真正值得反复琢磨的是：

- 顶尖 AI 产品团队如何理解「工具」和「系统」的区别
- 为什么他们越来越依赖原型而不是文档
- 为什么多智能体的关键不是 agent 数量，而是上下文结构
- 为什么未来的竞争力，越来越来自跨学科整合能力，而不是单一技术熟练度

如果把这些内容放在一起看，Boris 传递出的其实是一个非常清晰的信号：

> 我们已经不在「AI 会不会改变工作方式」的阶段了。
>
> 问题变成了：**你准备怎么在这种变化里重建自己的工作方式。**
